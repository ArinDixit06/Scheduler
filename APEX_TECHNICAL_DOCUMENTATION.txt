1. OVERVIEW & VISION
================================================================================

APEX is an ultimate personal productivity application designed to unify task
management, calendar scheduling, habit tracking, deep work focus, and AI-powered
assistance into a single coherent experience.

Core Design Principles:
  - Zero Friction: Capture anything in under 2 seconds
  - One Source of Truth: No context-switching between apps
  - AI-First: Intelligence is woven into every feature, not bolted on
  - Wellbeing-Aware: The app tracks energy and prevents overload, not just output
  - Offline-Capable: Every core feature works without a connection

Target Platforms:
  - iOS 15+
  - Android 11+
  - (Future) Web via React Native Web

2. SYSTEM ARCHITECTURE
================================================================================

High-Level Architecture:

  [React Native App]
       |
       | HTTPS / WSS
       |
  [Express API Server]  ←→  [Redis (Cache + Queues)]
       |
       |──→ [PostgreSQL (Primary DB via Prisma)]
       |──→ [AI Microservice (Claude API)]
       |──→ [BullMQ Workers]
       |         |──→ Calendar Sync Worker
       |         |──→ AI Insights Worker
       |         |──→ Push Notification Worker
       |         └──→ Email Processing Worker
       |──→ [Socket.IO Server]
       |──→ [AWS S3 (File Storage)]
       └──→ [Firebase FCM (Push)]

External Integrations:
  [Google APIs]         — Calendar, Gmail, OAuth
  [Microsoft Graph]     — Outlook Calendar, Teams
  [Slack API]           — Messages, Slash Commands
  [GitHub API]          — Issues, PRs
  [Anthropic API]       — Claude AI (all AI features)

Data Flow Summary:
  1. Mobile app authenticates → receives JWT access + refresh tokens
  2. All API calls go through /api/* with Bearer token auth
  3. Real-time updates arrive via Socket.IO (/user namespace)
  4. Background jobs run on BullMQ workers, results pushed via Socket.IO
  5. AI features call Claude API through the ai-service package
  6. File uploads go directly to S3 via pre-signed URLs

  Forms           : react-hook-form + zod resolvers
  Icons           : @expo/vector-icons (MaterialCommunityIcons)

BACKEND (apps/backend)
  Runtime         : Node.js 20 LTS
  Framework       : Express 4.19
  Language        : TypeScript 5.4
  ORM             : Prisma 5 (with Prisma Migrate)
  Database        : PostgreSQL 16
  Cache           : Redis 7 (ioredis)
  Job Queue       : BullMQ 5

  Provider        : Anthropic Claude API (claude-sonnet-4-20250514)

  Streaming       : Server-Sent Events (SSE) for chat

INFRASTRUCTURE

  CI/CD           : GitHub Actions
  Hosting         : AWS EC2 (API) + RDS (PostgreSQL) + ElastiCache (Redis)
  CDN             : CloudFront (S3 assets)

6. API REFERENCE
================================================================================

Base URL: https://api.apexapp.io/api
Auth: Bearer <accessToken> on all routes except /auth/*

All responses follow this envelope:
  Success: { success: true, data: <payload> }
  Error:   { success: false, error: { code: string, message: string } }

Pagination (list endpoints):
  Query params: ?page=1&limit=20
  Response includes: { data: [], meta: { page, limit, total, totalPages } }

--- AUTH ---

POST /auth/register
  Body: { email, password, name, timezone }
  Returns: { user, accessToken, refreshToken }

POST /auth/login
  Body: { email, password }
  Returns: { user, accessToken, refreshToken }

POST /auth/refresh
  Body: { refreshToken }
  Returns: { accessToken, refreshToken }

POST /auth/logout
  Body: { refreshToken }
  Returns: { success: true }

POST /auth/oauth/google
  Body: { code, redirectUri }
  Returns: { user, accessToken, refreshToken }

--- TASKS ---

GET /tasks
  Query: status, projectId, tag, dueBefore, dueAfter, sort, page, limit
  Returns: paginated Task[]

POST /tasks
  Body: { title, description?, dueDate?, estimatedMinutes?, projectId?,
          tags?, manualPriority? }
  Returns: Task (AI scoring runs async)

GET /tasks/:id
  Returns: Task + subtasks[] + comments[] + attachments[]

PATCH /tasks/:id
  Body: partial Task fields
  Returns: updated Task

DELETE /tasks/:id
  Returns: { success: true }

POST /tasks/:id/complete
  Body: { actualMinutes? }
  Returns: updated Task

GET /tasks/ai/next-action
  Returns: { task: Task, reasoning: string }

POST /tasks/ai/bulk-schedule
  Body: { weekStart: ISODate }
  Returns: { scheduled: { taskId, slot }[], message: string }

--- CALENDAR ---

GET /calendar/events
  Query: start (ISO), end (ISO), sources? (comma-separated)
  Returns: CalendarEvent[]

POST /calendar/events
  Body: { title, startAt, endAt, description?, location?, meetingUrl?,
          allDay?, attendees? }
  Returns: CalendarEvent

POST /calendar/smart-schedule
  Body: { title, durationMinutes, deadline?, preferPeakEnergy? }
  Returns: { slots: [{ start, end, score, reason }] }

GET /calendar/events/:id/prep
  Returns: { agenda, context, actionItems, suggestedQuestions }
  (generates via AI if not cached)

--- HABITS ---

GET /habits
  Returns: Habit[] with today's entry status and streak

POST /habits
  Body: { title, frequency, customDays?, targetCount?, color?, icon?,
          reminderTime? }
  Returns: Habit

POST /habits/:id/log
  Body: { date, completed, count?, note? }
  Returns: HabitEntry + updated streak

GET /habits/:id/history

7. AI SERVICE LAYER

Location: packages/ai-service/src/

All AI functions accept a userId. Before calling Claude, each function fetches
the minimal necessary context from the DB to personalize the response.

--- FUNCTION CATALOG ---

CopilotChat(userId, messages, stream)
  Context fetched: user name, timezone, today's tasks (top 10 by priority),
                   today's events, current energy score, top 3 projects
  System prompt template:
    "You are APEX, a personal productivity AI for {name}.
     Current time: {localTime} ({timezone}).
     Today's meetings: {events}.
     Priority tasks: {tasks}.
     Energy level: {energyScore}/5.
     Be concise, direct, and action-oriented. When the user asks to change
     their schedule, confirm before acting."
  Returns: streaming SSE or full string

ScoreTaskPriority(task, userContext)
  Scoring factors:
    - Deadline urgency    (40% weight): hours until due / working hours available
    - Project importance  (25% weight): project priority rank
    - Effort vs capacity  (20% weight): estimated minutes vs free time today
    - User habits         (15% weight): time of day they usually do this type
  Returns: { score: 0-100, reasoning: string, suggestedTimeBlock: string }

GenerateMeetingPrepBrief(event, userId)
  Context: attendee email threads (last 3), previous meeting notes with same
           attendees, open tasks linked to the meeting topic
  Returns: { agenda[], context, actionItems[], suggestedQuestions[] }

SummarizeMeetingTranscript(transcript, attendees)
  Prompt strategy: structured JSON output requested
  Returns: { summary, actionItems[{ owner, task, deadline? }], decisions[],
             followUpDate? }

DetectOverload(userId, weekStart)
  Analysis: sum(estimatedMinutes for tasks due this week) vs
            (workingHoursPerDay * 5 * 60 * 0.7) [70% utilization cap]
  Returns: { overloaded, severity: LOW|MEDIUM|HIGH, tasksToDefer[],
             meetingLoad: number, message }

GenerateWeeklyRecap(userId, weekEnd)
  Aggregates: tasks completed, focus minutes, habit streaks, energy avg, wins
  Returns: { tasksCompleted, focusHours, habitStreaks, energyAvg, topWin,
             nextWeekSuggestions[], message }

SmartSchedule(userId, item)
  Fetches: free/busy slots from all calendars + blocked focus times
  Scores slots by: proximity to energy peak, buffer before/after meetings,
                   deadline pressure
  Returns: { suggestedSlots: [{ start, end, score, reason }][] }

BreakdownTask(task)
  Prompt: Ask Claude to decompose task into 3-7 concrete subtasks with durations
  Returns: { subtasks: [{ title, estimatedMinutes }], totalMinutes }

DraftEmailReply(emailContext, userContext)
  Context: user's writing style (sampled from past emails if available)
  Returns: { draft, tone: FORMAL|CASUAL, confidence: 0-1 }

GenerateProactiveInsights(userId)
  Runs nightly. Checks:
    - Tasks overdue > 2 days          → OVERLOAD insight
    - Habit streak at risk (missed yesterday) → PATTERN insight
    - 3+ back-to-back meetings tomorrow  → SUGGESTION (buffer time)
    - User completed 5+ tasks in a day   → SUGGESTION (positive momentum)
    - No deep work blocks this week      → SUGGESTION
  Writes AIInsight records, triggers push notification for HIGH priority

--- ERROR HANDLING ---

All AI functions implement retry logic:
  - 3 retries with exponential backoff (1s, 2s, 4s)
  - If Claude API unavailable: return cached insight if available,
    else return graceful degraded response (no AI, show raw data)

Rate limiting: 100 requests/user/hour to AI endpoints

13. AUTHENTICATION & SECURITY
================================================================================

--- JWT STRATEGY ---

Access Token:
  Algorithm: RS256 (asymmetric, 2048-bit key pair)
  Expiry: 15 minutes
  Payload: { sub: userId, email, iat, exp }
  Storage: in-memory (Zustand), NOT in AsyncStorage

Refresh Token:
  Algorithm: HS256
  Expiry: 30 days
  Storage (mobile): Expo SecureStore
  Storage (server): hashed in RefreshSession table
  Rotation: new refresh token issued on every use

--- PASSWORD SECURITY ---

Hashing: bcrypt (cost factor 12)
Reset flow:
  1. POST /auth/forgot-password → generate signed 1-hour token, email link
  2. POST /auth/reset-password  → verify token, update hash, invalidate all
     existing refresh sessions

--- OAUTH TOKENS ---

All third-party access/refresh tokens encrypted at rest:
  Algorithm: AES-256-GCM
  Key: APP_ENCRYPTION_KEY from environment
  IV: random 16 bytes, stored alongside ciphertext

--- RATE LIMITING ---

express-rate-limit config:
  /auth/*          : 10 requests / 15 minutes per IP
  /ai/chat         : 60 requests / hour per user
  /api/*           : 200 requests / minute per user
  Global           : 1000 requests / minute per IP

--- HTTP SECURITY HEADERS (Helmet.js) ---

Content-Security-Policy: default-src 'self'
X-Frame-Options: DENY
X-Content-Type-Options: nosniff
Referrer-Policy: strict-origin-when-cross-origin
Strict-Transport-Security: max-age=31536000; includeSubDomains

--- INPUT VALIDATION ---

All request bodies validated with Zod schemas before hitting controllers.
File uploads: mime-type verification + 10MB size limit.
SQL injection: not possible (Prisma parameterized queries).

--- GDPR ---

Data export: GET /user/me/export → returns JSON of all user data
Account deletion: DELETE /user/me
  → soft-deletes user, queues full cascade deletion job (30-day delay)
  → removes all OAuth tokens immediately
  → sends confirmation email

14. OFFLINE SUPPORT
================================================================================

Strategy: Offline-first with sync-on-reconnect

--- WHAT WORKS OFFLINE ---

  Full functionality:  Tasks (create, edit, complete), Habits (log entry),
                       Focus sessions (start, run, end), Quick Capture
  Read-only cached:    Calendar events, Projects, AI Insights (last fetched)
  Unavailable:         AI Copilot, Calendar sync, Integration actions

--- OFFLINE QUEUE ---

Implementation: Zustand appStore.offlineQueue (persisted via AsyncStorage)

Queue item shape:
  { id: uuid, method: 'POST'|'PATCH'|'DELETE', url: string,
    body: object, timestamp: number, retries: number }

On mutation while offline:
  1. Apply optimistically to React Query cache
  2. Push to offlineQueue in appStore
  3. Show "saved locally" indicator

On network reconnect (NetInfo event):
  1. Process queue in order (FIFO)
  2. For each item: make API call
  3. On success: remove from queue, resolve local state
  4. On 409 conflict: prompt user to resolve
  5. On persistent failure: show error with retry option

Conflict resolution:
  - Server timestamp wins for CalendarEvents
  - Last-write-wins for Task edits (by updatedAt)
  - Habit entries: client always wins (user intent)

19. PERFORMANCE GUIDELINES
================================================================================

--- BACKEND ---

Database:
  - Index all foreign keys (Prisma does this automatically)
  - Composite index on Task(userId, status, dueDate) for list queries
  - Index on CalendarEvent(userId, startAt, endAt) for date range queries
  - Use select in Prisma queries — never fetch all fields for list views
  - Pagination required on all list endpoints (max 50 per page)

Caching (Redis):
  - Cache today's events per user: TTL 5 minutes (key: calendar:userId:YYYY-MM-DD)
  - Cache AI priority scores: TTL 1 hour (key: task:priority:taskId)
  - Cache user context for AI: TTL 2 minutes (key: ai:context:userId)

Response targets:
  - List endpoints: < 150ms (p95)
  - Task create (without AI): < 100ms
  - AI chat first token: < 800ms
  - Calendar sync: < 30s per user

--- MOBILE ---

Render performance:
  - Use FlashList (Shopify) instead of FlatList for task/calendar lists
  - Memoize task cards (React.memo + stable key props)
  - Avoid anonymous functions in render (useCallback for handlers)
  - Use Reanimated worklets for timer — stays on UI thread
  - Images: use Expo Image with caching + blur hash placeholders

Bundle size:
  - Code-split by screen using React Navigation's lazy loading
  - Avoid importing entire icon sets — use tree-shakeable imports

Battery / network:
  - Background calendar sync: only when app in foreground or on Wi-Fi
  - Socket.IO: reconnect with exponential backoff (avoid hammering server)
  - React Query: staleTime=60s to prevent over-fetching
